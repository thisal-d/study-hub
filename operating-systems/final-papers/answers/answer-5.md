## Question 1

**a.**

**i. Binary Semaphore** — A semaphore restricted to integer values 0 and 1. It works like a mutex lock: value 1 means the resource is available; value 0 means it is locked. Used to provide mutual exclusion — only one process can be in the critical section at a time.

**ii. Signal Operation** — The `signal()` operation (also called V or up) on a semaphore increments its value by 1. If any process is blocked waiting on this semaphore, one is unblocked and moved from the waiting queue to the ready queue. It is the operation used to release a lock or notify a waiting process.

---

**b. Three requirements for a critical section solution:**

1. **Mutual Exclusion** — Only one process executes in its critical section at a time.
2. **Progress** — If no process is in its critical section, processes not in their remainder section can participate in deciding who enters next without indefinite postponement.
3. **Bounded Waiting** — A limit exists on how many times other processes can enter the critical section before a waiting process is granted access.

---

**c. Two atomic hardware instructions:**

1. Test-and-Set (`TestAndSet`)
2. Swap (`Swap` / Compare-and-Swap)

---

**d.**

**i. Critical Section of the ATM transaction system:**

The critical section is the code that reads and modifies the shared `account_balance` and writes to `TransactionLog`:

```
read account_balance            ← start of critical section
check balance >= withdraw_amt
account_balance = balance - withdraw_amt
dispense_cash()
write record to TransactionLog  ← end of critical section
```

Concurrent execution by multiple ATMs without synchronization can allow a customer to withdraw more than the account balance (a classic race condition called a lost update).

**ii. Solution using a Semaphore:**

```
Semaphore mutex = 1;     // binary semaphore

ATM_Process:
    // Entry Section
    wait(mutex);

    // Critical Section
    local_bal = account_balance;
    if (local_bal >= withdraw_amt):
        account_balance = local_bal - withdraw_amt;
        dispense_cash();          // write to TransactionLog

    // Exit Section
    signal(mutex);

    // Remainder Section
    ... (print receipt, update screen, etc.)
```

Only one ATM processes a withdrawal at a time, preventing concurrent balance corruption.

## Question 2

**a.**

**i. Compile time address binding** — The compiler generates absolute memory addresses if the memory location where the program will be loaded is known at compile time. The program must always be placed at the same location; if it moves, recompilation is required.

**ii. Base register** — A hardware register holding the starting physical address of a process's memory partition. Every logical address generated by the CPU is added to the base register value by the MMU to compute the physical address. It enables process relocation in memory without recompilation.

---

**b.**

Free segments: A=350KB, B=150KB, C=500KB, D=100KB. New process = 400KB.

**i. Two main problems in contiguous memory allocation:**

1. **External Fragmentation** — Total free memory (350+150+500+100=1100KB) exceeds the request but holes are scattered; no single hole may be large enough.
2. **Internal Fragmentation** — Allocated partition may exceed the actual need, wasting the excess portion.

**ii.**

**1. If compaction is used:**
All processes are relocated so free holes merge into one contiguous block of 1100KB. The 400KB process is allocated. *(Requires runtime relocation — high overhead.)*

**2. If paging is used:**
The 400KB process is divided into pages. Assuming page size = 100KB, the process needs 4 pages. Any 4 free frames anywhere in memory are allocated. A page table maps the 4 logical pages to their physical frames.

**iii.** Modern OSes use **paging** — it avoids external fragmentation and does not require process relocation.

**iv.** A page table maps the logical page number to the physical frame number. Since paging places pages non-contiguously, the MMU cannot compute the physical address without this translation table.

---

**c.** Physical = 256MB = 2²⁸ bytes | Virtual = 2GB = 2³¹ bytes | Page = 32KB = 2¹⁵ bytes

**i. Number of pages** = 2³¹ / 2¹⁵ = **2¹⁶ = 65,536 pages**

**ii. Number of frames** = 2²⁸ / 2¹⁵ = **2¹³ = 8,192 frames**

**iii. Bit allocations:**
- Offset = 15 bits
- Page number = 16 bits
- Frame number = 13 bits

**iv. Page table entries** = **65,536 entries**

## Question 3

**a. Four necessary conditions for deadlock:**

1. **Mutual Exclusion** — At least one resource is non-sharable.
2. **Hold and Wait** — A process holds one resource and waits for more.
3. **No Pre-emption** — Resources cannot be forcibly taken from a process.
4. **Circular Wait** — A circular chain of processes, each waiting for a resource held by the next.

---

**b. Two solutions to prevent deadlock by denying No Pre-emption:**

1. **Forcible preemption** — If a process holding resources requests another that is unavailable, all resources currently held by that process are preempted (released). The process is restarted only when all required resources are available.
2. **Pre-emption of victim's resources** — If process P requests a resource held by process Q, and Q is waiting (not running), preempt Q's resource and give it to P. Q restarts when it can reacquire all its resources.

*(Note: This works well only for resources whose state can be saved and restored, such as CPU registers and memory pages.)*

---

**c.** Modern general-purpose OSes **ignore the deadlock problem** (Ostrich Algorithm). The cost of avoidance/prevention outweighs the benefit, as deadlocks are rare. Recovery is done by user intervention (killing processes or rebooting).

---

**d.**

## Question 4

**a.**

**i. Demand Paging** — A lazy-loading strategy where pages are only brought into physical memory when they are accessed for the first time. The process starts executing with no pages loaded; page faults bring in pages as needed, minimising memory usage.

**ii. Valid / Invalid bit** — A bit in each page table entry. **Valid (1)** = the page is currently in a physical frame and accessible. **Invalid (0)** = the page is on disk (triggers a page fault) or the logical address is illegal.

---

**b. Steps on a page fault:**

1. Trap to OS; save process state.
2. Verify the memory reference is valid. If invalid, terminate the process.
3. Find a free frame. If none, select a victim using a page replacement algorithm.
4. If victim page is modified (dirty bit set), write it back to disk.
5. Load the required page from disk into the free frame via disk I/O.
6. Update the page table: record frame number and set valid bit = 1.
7. Restart the faulting instruction.

---

**c. Virtual Address vs Physical Address:**

| | Virtual Address | Physical Address |
|---|---|---|
| Generated by | CPU / user process | MMU hardware |
| Also called | Logical address | Real address |
| Space size | Can exceed physical RAM | Limited to installed RAM |
| User visibility | Visible to program | Hidden from program |
| Translation | Via page table (MMU/TLB) | Direct RAM access |

---

**d. Two hardware resources needed for virtual memory:**

1. **MMU (Memory Management Unit)** — translates virtual page numbers to physical frame numbers during execution.
2. **Secondary Storage (Backing Store / Disk)** — stores pages not currently in physical memory. A **TLB** (Translation Lookaside Buffer) is also needed for fast address translation caching.
