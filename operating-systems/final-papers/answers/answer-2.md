## Question 1

**a.**

**i. Race Condition** — A situation where two or more processes access shared data concurrently and the final outcome depends on the relative order of execution. The result is non-deterministic and can lead to data corruption. For example, if two booking counters read `available_seats` at the same time and both decrement it, a seat may be double-booked.

**ii. Spinlock** — A synchronization mechanism where a process repeatedly checks (spins in a loop) whether a lock is available, instead of being put to sleep. It is efficient when the wait time is very short but wastes CPU cycles if the lock is held for long periods.

---

**b. Three requirements for a critical section solution:**

1. **Mutual Exclusion** — Only one process may execute in its critical section at a time.
2. **Progress** — If no process is in the critical section, the decision of which process enters next cannot be postponed indefinitely.
3. **Bounded Waiting** — A bound must exist on how many times other processes can enter the critical section before a waiting process is granted entry.

---

**c. Two atomic hardware instructions:**

1. Test-and-Set (`TestAndSet`)
2. Swap (`Swap` / Compare-and-Swap)

---

**d.**

**i. Critical Section of the cinema booking system:**

The critical section is the code that reads and modifies the shared variable `available_seats` and writes the confirmed booking to `SeatReservationDB`:

```
read available_seats          ← start of critical section
decrement available_seats
confirm and store booking     ← end of critical section
```

All three operations together form the critical section because interruption between them causes race conditions.

**ii. Solution using a Semaphore:**

```
Semaphore mutex = 1;    // binary semaphore, initialized to 1

Booking_Counter_Process:
    // Entry Section
    wait(mutex);

    // Critical Section
    local = available_seats;
    if (local > 0):
        available_seats = local - 1;
        confirm_booking();      // write to SeatReservationDB

    // Exit Section
    signal(mutex);

    // Remainder Section
    ... (display confirmation to customer, etc.)
```

Only one booking counter can modify `available_seats` at a time, preventing double-booking.

## Question 2

**a.**

**i. Load time address binding** — Address binding is performed at load time. If the location of a program in memory is not known at compile time, relocatable code is generated by the compiler. When the program is loaded into memory, the loader converts all relocatable addresses into absolute physical addresses. If the starting address changes, the program must be reloaded.

**ii. Limit register** — A hardware register that stores the length (size) of a process's memory partition. The CPU checks every logical address against the limit register. If the logical address is greater than or equal to the limit, a trap (addressing error) is raised, protecting one process's memory from being accessed by another.

---

**b.**

Free segments: A=200KB, B=500KB, C=100KB, D=400KB. New process = 350KB.

**i. Two main problems in contiguous memory allocation:**

1. **External Fragmentation** — Free memory exists in total but is scattered across non-contiguous holes.
2. **Internal Fragmentation** — Allocated partition may be larger than needed, wasting space inside the block.

**ii.**

**1. If compaction is used:**
All occupied processes are shifted to one end of memory, merging all free holes. Total free = 200+500+100+400 = 1200KB. After compaction, a single 1200KB free block exists. The 350KB process is allocated from this block. *(Note: compaction requires all processes to be relocatable and has high runtime overhead.)*

**2. If paging is used:**
The 350KB process is split into pages. Assuming page size = 50KB, the process needs 7 pages. The OS allocates any 7 free frames from available memory (they do not need to be contiguous). A page table maps each of the 7 logical pages to their physical frames.

**iii.** Modern operating systems use **paging**. Compaction is too expensive in terms of time and is generally not used in general-purpose OSes.

**iv.** A page table translates a process's logical page numbers to physical frame numbers. Since pages are scattered non-contiguously in RAM, the CPU cannot directly compute the physical address without the page table mapping.

---

**c.** Physical = 512MB = 2²⁹ bytes | Virtual = 4GB = 2³² bytes | Page = 8KB = 2¹³ bytes

**i. Number of pages** = 2³² / 2¹³ = **2¹⁹ = 524,288 pages**

**ii. Number of frames** = 2²⁹ / 2¹³ = **2¹⁶ = 65,536 frames**

**iii. Bit allocations:**
- Offset = 13 bits
- Page number = 19 bits
- Frame number = 16 bits

**iv. Page table entries** = Number of pages = **524,288 entries**

## Question 3

**a. Four necessary conditions for deadlock:**

1. **Mutual Exclusion** — At least one resource must be non-sharable.
2. **Hold and Wait** — A process holds a resource and waits for more.
3. **No Pre-emption** — Resources cannot be forcibly taken away.
4. **Circular Wait** — A circular chain of processes each waiting for a resource held by the next.

---

**b. Two solutions to prevent deadlock by denying Circular Wait:**

1. **Impose a total ordering on resources** — Assign a unique number to each resource type. Processes must always request resources in strictly increasing order of number. This prevents circular chains from forming.
2. **Two-phase locking** — A process must acquire all needed resources in a fixed global order (phase 1), then release them all (phase 2). No process can request a lower-numbered resource while holding a higher-numbered one.

---

**c.** Modern general-purpose OSes typically **ignore the deadlock problem** (Ostrich Algorithm). Prevention and avoidance have too much overhead for general use. Deadlocks are assumed to be rare and are resolved by user intervention (killing processes) or rebooting.

---

## Question 4

**a.**

**i. Thrashing** — A condition where a process spends more time paging (swapping pages in and out) than executing useful work. It occurs when a process does not have enough frames, causing frequent page faults. The CPU utilization drops dramatically as the OS is constantly handling page faults.

**ii. Page Buffering Algorithm** — An optimization where the OS maintains a pool of free frames. When a page fault occurs, the needed page is loaded into a free frame from the pool (the process restarts immediately), while the victim page is written back to disk in the background. This reduces the time a process waits during a page fault.

---

**b. Steps taken by OS on a page fault:** (same as Paper 1)

1. Check internal table to determine if the reference is valid or invalid.
2. If invalid, terminate. If valid but not in memory, continue.
3. Find a free frame from the free-frame list.
4. Schedule disk I/O to load the required page into the free frame.
5. Update the page table: set frame number and valid bit to 1.
6. Restart the instruction that caused the page fault.

---

**c. Virtual Address vs Physical Address:**

| | Virtual Address | Physical Address |
|---|---|---|
| Generated by | CPU / user process | MMU hardware |
| Also known as | Logical address | Real address |
| Size | May exceed physical memory | Limited to installed RAM |
| Visibility | Visible to process | Visible to hardware |
| Translation | Mapped by MMU via page table | Used directly to access RAM |

---

**d. Two hardware resources needed for virtual memory:**

1. **MMU (Memory Management Unit)** — translates virtual addresses to physical addresses using the page table.
2. **Secondary Storage (Disk / Backing Store)** — holds pages not currently in physical memory; also a **TLB** (Translation Lookaside Buffer) for fast lookup.
