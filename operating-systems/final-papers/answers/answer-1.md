## Question 1

**a.**

**i. Critical Section** — A segment of code in which a process accesses shared resources (e.g., shared variables, files, hardware). Only one process is permitted to execute inside its critical section at a time to prevent data inconsistency.

**ii. Semaphore** — An integer variable used for process synchronization. It is accessed only through two atomic operations: `wait()` (also called P or down) and `signal()` (also called V or up). A binary semaphore has value 0 or 1; a counting semaphore can take any non-negative integer value.

---

**b. Three requirements for a critical section solution:**

1. **Mutual Exclusion** — If process Pi is executing in its critical section, no other process may execute in its critical section.
2. **Progress** — If no process is executing in its critical section and some processes wish to enter, only those not in their remainder section can participate in deciding which enters next, and this decision cannot be postponed indefinitely.
3. **Bounded Waiting** — A bound must exist on the number of times other processes are allowed to enter their critical sections after a process has made a request to enter its critical section, before that request is granted.

---

**c. Two atomic hardware instructions:**

1. Test-and-Set (`TestAndSet`)
2. Swap (`Swap` / Compare-and-Swap)

**d.**

**i. Critical Section of the attendance system:**

The critical section is the code segment that reads and writes to the shared `EmpAttendaceLog` file. Specifically:

```
read current log state
write attendance record to EmpAttendaceLog
```

Any code that reads or modifies the shared log file is the critical section, because concurrent access by multiple department systems can lead to race conditions.

**ii. Solution using a Semaphore:**

```
Semaphore mutex = 1;   // initialized to 1 (binary semaphore)

Attendance_Recording_Process:
    // Entry Section
    wait(mutex);

    // Critical Section
    read EmpAttendaceLog;
    write attendance record to EmpAttendaceLog;

    // Exit Section
    signal(mutex);

    // Remainder Section
    ... (other non-shared operations)
```

Only one department system can write to EmpAttendaceLog at a time. All others block at `wait(mutex)` until the mutex is released.

## Question 2

**a.**

**i. Compile time address binding** — If the memory location where a process will reside is known at compile time, then the compiler generates absolute (physical) addresses directly in the code. The process must always be loaded at the same memory location. If the location changes, the program must be recompiled.

**ii. Base register** — A hardware register that holds the starting physical address of the currently running process. The CPU adds the value in the base register to every logical address generated by the process to produce the corresponding physical address, enabling relocation of processes in memory.

---

**b.**

Free segments: A=300KB, B=100KB, C=200KB, D=100KB. New process = 400KB.

**i. Two main problems in contiguous memory allocation:**

1. **External Fragmentation** — Total free memory is sufficient to satisfy a request, but the free spaces are not contiguous (scattered), so no single hole is large enough.
2. **Internal Fragmentation** — Allocated memory partition may be slightly larger than the requested memory, wasting the unused portion inside the allocated block.

**ii.**

**1. If compaction is used:**
The OS relocates all currently loaded processes so their memory partitions are contiguous, merging all free holes into one large free block. Total free memory = 300+100+200+100 = 700KB. After compaction, there is one contiguous free block of 700KB, which is large enough for the 400KB process to be allocated.

**2. If paging is used:**
The 400KB process is divided into equal-sized pages. Assuming page size = 100KB, the process requires 4 pages. The OS allocates any 4 free frames from memory (frames do not need to be contiguous). A page table is created to map each logical page to its corresponding physical frame. The process runs without needing contiguous physical memory.

**iii.** Paging is used by modern operating systems (e.g., Linux, Windows). Compaction is not practical because relocating all processes at runtime has very high overhead.

**iv.** A page table is needed because in paging, a process's logical pages are mapped to non-contiguous physical frames. When the CPU generates a logical address (page number + offset), the page table translates the page number to the correct frame number in physical memory so the correct physical address can be computed.

**c.** Physical = 256MB = 2²⁸ bytes | Virtual = 2GB = 2³¹ bytes | Page = 16KB = 2¹⁴ bytes

**i. Number of pages** = Virtual / Page size = 2³¹ / 2¹⁴ = **2¹⁷ = 131,072 pages**

**ii. Number of frames** = Physical / Page size = 2²⁸ / 2¹⁴ = **2¹⁴ = 16,384 frames**

## Question 3

**a. Four necessary conditions for deadlock (Coffman Conditions):**

1. **Mutual Exclusion** — At least one resource is held in a non-sharable mode; only one process can use it at a time.
2. **Hold and Wait** — A process holds at least one resource and is waiting to acquire additional resources held by other processes.
3. **No Pre-emption** — Resources cannot be forcibly taken from a process; they must be released voluntarily.
4. **Circular Wait** — A set of processes {P₀, P₁, …, Pₙ} exists such that P₀ waits for a resource held by P₁, P₁ waits for P₂, …, Pₙ waits for P₀.

---

**b. Two solutions to prevent deadlock by denying Hold and Wait:**

1. **Require all resources upfront** — A process must request and be allocated all resources it will need before it begins execution. If any resource is unavailable, the process must wait without holding any.
2. **Release before new request** — A process may request resources only when it currently holds none. If it needs additional resources, it must release all currently held resources first, then re-request everything it needs.

---

**c. How modern general-purpose OS solves deadlock:**

Modern general-purpose operating systems (e.g., Linux, Windows) typically **ignore the deadlock problem** — this is known as the **Ostrich Algorithm**. The rationale is that deadlocks are rare, and the performance cost of prevention or avoidance (e.g., Banker's Algorithm) is too high for general use. If a deadlock occurs, the system may rely on the user to detect it (e.g., killing an unresponsive process or rebooting).

## Question 4

**a.**

**i. Demand Paging** — A method of virtual memory management where pages of a process are only loaded into physical memory when they are demanded (accessed), not all at once at load time. This reduces memory usage and allows processes larger than physical memory to run.

**ii. Valid / Invalid bit** — A bit associated with each page table entry. A **valid** bit (1) means the page is currently loaded in a physical frame and the address is legal. An **invalid** bit (0) means the page is either not currently in memory (causing a page fault when accessed) or the address is outside the process's logical address space.

---

**b. Steps taken by OS when a page fault occurs:**

1. Check an internal table (usually kept with the PCB) to determine whether the reference was a valid or invalid memory access.
2. If invalid, terminate the process. If valid but not in memory, continue.
3. Find a free frame (from the free-frame list).
4. Schedule a disk I/O operation to bring the required page from backing store into the free frame.
5. When the disk read completes, update the page table entry: set frame number and change the valid bit to valid (1).
6. Restart the instruction that caused the page fault. The process can now access the page.

---

**c. Virtual Address vs Physical Address:**

| | Virtual Address | Physical Address |
|---|---|---|
| **Definition** | Address generated by the CPU during program execution | Actual address in physical memory (RAM) |
| **Also called** | Logical address | Real address |
| **Generated by** | CPU / user program | Memory Management Unit (MMU) |
| **Visible to** | User/process | Hardware (RAM) |
| **Range** | Can exceed physical memory size | Limited to actual installed RAM |
| **Translation** | Converted to physical by MMU using page table | Used directly to access RAM |

---

**d. Two hardware resources needed for virtual memory:**

1. **Page Table** (hardware support via MMU — Memory Management Unit) — translates logical page numbers to physical frame numbers.
2. **Secondary Storage (Disk / Backing Store)** — stores pages not currently in physical memory. Also requires a **TLB (Translation Lookaside Buffer)** as a hardware cache for fast page table lookups.
