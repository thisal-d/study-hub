## Question 1

**a.**

**i. Mutex Lock** — A mutual exclusion lock. A binary variable that can be either locked or unlocked. A process must acquire (lock) the mutex before entering the critical section and release (unlock) it on exit. Only the process that locked it can unlock it, ensuring exclusive access.

**ii. Monitor** — A high-level synchronization construct that encapsulates shared variables, the procedures that operate on them, and the synchronization between concurrent processes. Only one process can be active inside a monitor at a time. Monitors use **condition variables** (with `wait()` and `signal()`) to allow processes to wait for conditions to become true.

---

**b. Three requirements for a critical section solution:**

1. **Mutual Exclusion** — Only one process executes in its critical section at a time.
2. **Progress** — Processes not in their remainder section decide which enters next; this cannot be delayed indefinitely.
3. **Bounded Waiting** — A limit exists on how many times other processes enter before a waiting process is granted access.

---

**c. Two atomic hardware instructions:**

1. Test-and-Set (`TestAndSet`)
2. Swap (`Swap` / Compare-and-Swap)

**d.**

**i. Critical Section of the blood inventory system:**

The critical section is the code that reads and modifies the shared variable `blood_units` and writes to the shared `BloodInventoryLog`:

```
read blood_units              ← start of critical section
check if sufficient units available
decrement blood_units
write record to BloodInventoryLog  ← end of critical section
```

This entire read-modify-write block is the critical section. Concurrent access by 8 wards without synchronization can cause the same blood unit to be issued to two patients simultaneously.

**ii. Solution using a Semaphore:**

```
Semaphore mutex = 1;    // binary semaphore

Ward_Process:
    // Entry Section
    wait(mutex);

    // Critical Section
    local = blood_units;
    if (local >= required_units):
        blood_units = local - required_units;
        issue_blood();          // write to BloodInventoryLog

    // Exit Section
    signal(mutex);

    // Remainder Section
    ... (update patient records, etc.)
```

The semaphore ensures only one ward accesses the blood inventory at a time.

## Question 2

**a.**

**i. Execution time address binding** — Address binding is deferred until run time. The process can be moved from one memory segment to another during execution. This requires hardware support (base and limit registers). The logical address is converted to a physical address by the MMU at the time the instruction executes.

**ii. Base register** — A hardware register storing the starting physical address of a process's memory partition. Every logical address generated by the CPU is added to the base register value by the MMU to produce the physical address. Enables process relocation without recompilation.

---

**b.**

Free segments: A=150KB, B=400KB, C=250KB, D=200KB. New process = 300KB.

**i. Two main problems in contiguous memory allocation:**

1. **External Fragmentation** — Total free memory (150+400+250+200=1000KB) exists but no single contiguous hole is large enough if scattered.
2. **Internal Fragmentation** — Allocated block may be slightly larger than requested, wasting the excess inside the partition.

**ii.**

**1. If compaction is used:**
The OS relocates all processes to one end, creating one large free block of 1000KB total. The 300KB process is allocated from this block. *(High overhead — all processes must be relocated.)*

**2. If paging is used:**
The 300KB process is divided into pages. Assuming page size = 50KB, the process needs 6 pages. The OS allocates any 6 free frames from anywhere in memory. A page table maps each logical page to its physical frame.

**iii.** Modern OSes use **paging** — it avoids external fragmentation without requiring all processes to be relocated.

**iv.** The page table is needed because paging places a process's pages into non-contiguous frames. The CPU generates logical (page number, offset) addresses; the page table translates the page number to the corresponding frame number so the correct physical address can be accessed.

---

**c.** Physical = 128MB = 2²⁷ bytes | Virtual = 1GB = 2³⁰ bytes | Page = 4KB = 2¹² bytes

**i. Number of pages** = 2³⁰ / 2¹² = **2¹⁸ = 262,144 pages**

**ii. Number of frames** = 2²⁷ / 2¹² = **2¹⁵ = 32,768 frames**

**iii. Bit allocations:**
- Offset = 12 bits
- Page number = 18 bits
- Frame number = 15 bits

**iv. Page table entries** = **262,144 entries**

## Question 3

**a. Four necessary conditions for deadlock:**

1. **Mutual Exclusion** — At least one resource is non-sharable.
2. **Hold and Wait** — A process holds a resource while waiting for more.
3. **No Pre-emption** — Resources cannot be forcibly removed.
4. **Circular Wait** — A circular chain of processes, each waiting for the next.

---

**b. Two solutions to prevent deadlock by denying Mutual Exclusion:**

1. **Use sharable resources wherever possible** — Design resources to be sharable (e.g., read-only files can be accessed by multiple processes simultaneously). If mutual exclusion is not required, it cannot contribute to deadlock.
2. **Spooling** — For resources that must be exclusive (e.g., printers), use a spooler process that queues requests. Only the spooler process accesses the device, so user processes never directly hold the resource — eliminating mutual exclusion for them.

*(Note: Mutual exclusion cannot be fully eliminated for all resource types, so this prevention method has limited applicability.)*

---

**c.** Modern general-purpose OSes **ignore the deadlock problem** (Ostrich Algorithm). Prevention and avoidance are too costly. Deadlocks are rare and handled by user intervention (killing a process or rebooting).

---

## Question 4

**a.**

**i. Demand Paging** — Pages of a process are loaded into physical memory only when they are actually referenced (demanded), not all at once at process startup. This allows large programs to run in limited physical memory.

**ii. Page Fault** — An interrupt/trap raised by the hardware when a process references a page whose valid bit is 0 (not currently in physical memory). The OS must handle the page fault by loading the required page from disk into a free frame.

---

**b. Steps on a page fault:**

1. Check if the memory reference is valid or invalid using the internal table.
2. If invalid, terminate the process. If valid but not in memory, proceed.
3. Find a free frame (from the free-frame list; if none, select a victim page using a replacement algorithm).
4. If a victim page is dirty (modified), write it to disk first.
5. Schedule a disk I/O to load the required page into the free frame.
6. Update the page table: set the frame number and change the valid bit to 1.
7. Restart the instruction that caused the page fault.

---

**c. Virtual Address vs Physical Address:**

| | Virtual Address | Physical Address |
|---|---|---|
| Generated by | CPU during execution | MMU hardware |
| Also called | Logical address | Real address |
| Max size | Can exceed physical RAM | Limited to installed RAM |
| User visibility | Visible to process | Hidden from process |
| Translation | Converted to physical via page table | Used directly to access RAM |

---

**d. Two hardware resources needed for virtual memory:**

1. **MMU (Memory Management Unit)** with page table support — translates logical to physical addresses.
2. **Secondary Storage (Backing Store / Disk)** — stores pages not currently in RAM; a **TLB** accelerates page table lookups.
